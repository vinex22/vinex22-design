# Cinematic Patterns · best practices for workflow demo animations

> Five key patterns for upgrading from "PowerPoint animation" to "keynote-grade cinematic".
> Distilled from two cinematic workflow demos — proven and reproducible.

---

## 0 · What this document solves

When you need to make a "demo animation showing a workflow" (typical scenarios: skill workflows, product onboarding, API call flows, agent task execution), you have two common approaches:

| Paradigm | What it looks like | Result |
|---|---|---|
| **PowerPoint animation** (bad) | step 1 fade in → step 2 fade in → step 3 fade in, 4 boxes side-by-side on screen | Audience feels "this is a PowerPoint with a fade effect" — no wow moment |
| **Cinematic** (good) | scene-based, focus one thing at a time, scenes connected by dissolve / focus pull / morph | Audience feels "this is a clip from a product launch" — they want to screenshot and share |

The difference isn't **animation technique**, it's **narrative paradigm**. This file covers how to upgrade from the former to the latter.

## 1 · Five core patterns

### Pattern A · Dashboard + Cinematic Overlay (two-layer structure)

**Problem**: a pure cinematic defaults to a black screen + a ▶ button — if the user lands on the page and doesn't click, they see nothing.

**Solution**:
```
DEFAULT state (always visible): full static workflow dashboard
  └── audience sees at a glance how this skill / workflow runs

POINT ▶ trigger (overlay rises): 22-second cinematic
  └── after playback, fades back to DEFAULT
```

**Implementation points**:
- `.dash` is visible by default; `.cinema` defaults to `opacity: 0; pointer-events: none`
- `.play-cta` is a small gold button bottom-right (not a giant central overlay)
- On click → `cinema.classList.add('show')` + `dash.classList.add('hide')`
- Use `requestAnimationFrame` to play once (not loop); after end call `endCinematic()` to reverse the state

**Anti-pattern**: default = central giant ▶ overlay covering everything; the page is blank before clicking.

---

### Pattern B · Scene-based, NOT Step-based

**Problem**: splitting the animation into "step 1 appears → step 2 appears → ..." is PowerPoint thinking.

**Solution**: split into 5 scenes; each scene is an **independent shot**, full-screen focused on one thing:

| Scene type | Job | Duration |
|---|---|---|
| 1 · Invoke | User input trigger (terminal typewriter) | 3–4s |
| 2 · Process | Visualization of the core workflow (unique visual language) | 5–6s |
| 3 · Result/Insight | Distilled key artifact (visualization) | 4–5s |
| 4 · Output | Actual output display (file / diff / numbers) | 3–4s |
| 5 · Hero Reveal | Closing hero moment (large type + value proposition) | 4–5s |

**Total ≈ 22 seconds** — this is the tested golden length:
- Shorter than 18 s: PMs aren't engaged before it ends
- Longer than 25 s: audience loses patience
- 22 s is enough for "hook → unfold → resolve → leave an impression"

**Implementation points**:
- `T = { DURATION: 22.0, s1_in: [0, 0.7], s2_in: [3.8, 4.6], ... }` global timeline
- A single `requestAnimationFrame(render)` runs all scene opacity / transform calcs
- Don't use setTimeout chains (easy to break, hard to debug)
- Easing must be `expoOut` / `easeOut` / cubic-bezier — **forbid linear**

---

### Pattern C · Each demo's visual language must be independent

**Problem**: after finishing the first cinematic, you cut corners on the second by reusing the same template (same orbit + pentagon + typewriter + hero giant text), only swapping copy.

**Result**: audience sees "these two skills look identical" — equivalent to saying "these two skills are no different".

**Solution**: each workflow's core metaphor differs, so the visual language must differ.

**Comparison case**:

| Dimension | "Distill people" workflow | "Optimize skill" workflow |
|---|---|---|
| Core metaphor | Collect → distill → write | Loop → evaluate → ratchet |
| Visual motion | Float / radiate / pentagon | Loop / climb / compare |
| Scene 2 | 3D Orbit · 8 archives floating in a perspective ellipse | Spin Loop · token runs 5 laps along a 6-node ring |
| Scene 3 | Pentagon · 5 tokens radiating from center | v1 vs v5 · side-by-side diff (red vs gold) |
| Scene 4 | Skill file typewriter | Hill-Climb · full-screen curve drawing |
| Scene 5 hero | "21 minutes" serif italic giant text | Rotating gear ⚙ + "KEPT +1.1" gold tag |

**Judgment criterion**: cover the copy and look only at visuals — can you tell which demo is which? If you can't, you cut corners.

---

### Pattern D · Use real materials generated by AI; not emoji or hand-drawn SVG

**Problem**: 3D orbit / gallery scenes need material fragments floating; emoji (📚🎤) are ugly and brand-less; hand-drawn SVG book spines never look like real books.

**Solution**: use a `gpt-image` model to generate one 4×2 grid image (8 thematic objects · white bg · 60px breathing space · unified style), then use `extract_grid.py --mode bbox` to crop into 8 transparent PNGs.

**Prompt essentials**:
- IP anchoring ("1960s archive aesthetic" / "consistent treatment across the set")
- White background (easy to crop; gray bg has mood but transparent extraction is hard)
- 4×2 not 5×5 (avoid last-row compression bug)
- Persona finishing ("You are a magazine curator preparing an exhibition photo")

**Anti-pattern**: emoji as icons; CSS silhouettes as product images.

---

### Pattern E · BGM + SFX dual track

**Problem**: animation without sound — audience subconsciously feels "this is a cheap demo".

**Solution**: BGM long-tone + 11 SFX cues.

**Generic SFX cue recipe** (works for workflow demos):

| Time | SFX | Trigger scene |
|---|---|---|
| 0.10s | whoosh | Terminal rises from below |
| 3.0s | enter | Typewriter completes, press enter |
| 4.0s | slide-in | Scene 2 elements enter |
| 5–9s × 5 | sparkle | Key process nodes (per generation / token / data point) |
| 14s | click | Switch to output scene |
| 17.8s | logo-reveal | Hero reveal moment |
| typewriter | type | Trigger every 2 chars (don't over-density) |

**Frequency separation**: BGM volume 0.32 (low-frequency floor); SFX volume 0.55 (mid/high punch); sparkle 0.7 (must be salient); logo-reveal 0.85 (the strongest hero moment).

**User control**:
- Must have a ▶ start overlay (browser autoplay restriction)
- Top-right small mute button (user can mute anytime)
- Don't make it "force-play when navigating to this page"

For full audio rules (frequency separation, ducking, BGM/SFX recipes), see [`audio-design-rules.md`](audio-design-rules.md). Audio binaries are not bundled — see [`sfx-library.md`](sfx-library.md) for prompts to regenerate them.

## 2 · Static Dashboard design points

The dashboard is Layer 1 of the two-layer structure. PMs who don't click ▶ should still understand the skill.

**Layout**: 3-column grid (or 1 large + 2 small); each panel solves one question:

| Panel type | What it answers | Example |
|---|---|---|
| **Pipeline / Flow Diagram** | "What is this skill's workflow?" | 4-stage pipeline · autoresearch loop |
| **Snapshot / State** | "What does the actual output data look like?" | 8-dimension rubric snapshot |
| **Trajectory / Evolution** | "How does it change over multiple runs?" | 5-generation hill-climb curve |
| **Examples / Gallery** | "What has been produced before?" | 21 personas gallery |
| **Strip · Example I/O** | "Input what → output what" | Example strip: `› distill feynman → feynman.skill (21 min)` |

**Key constraints**:
- Information density must be high (every panel carries differentiated info)
- But no data slop (every number must mean something)
- Color scheme matches cinematic (same family — switching isn't jarring)

## 3 · Debug & development tools

Any long animation needs three dev tools or debugging will explode.

### Tool 1 · `?seek=N` freeze at second N

```js
const seek = parseFloat(params.get('seek'));
if (!isNaN(seek)) {
  started = true; muted = true;
  frozenT = seek;  // render() uses this t instead of elapsed
  cinema.classList.add('show'); dash.classList.add('hide');
}

// In render():
let t = frozenT !== null ? frozenT : (elapsed % T.DURATION);
```

Usage: `http://.../slide.html?seek=12` jumps directly to second 12 — no waiting.

### Tool 2 · `?autoplay=1` skip the ▶ overlay

Convenient for Playwright auto-screenshot tests, and for force-starting when embedded in iframes.

### Tool 3 · Manual REPLAY button

Top-right small button — user / debugger can replay any number of times. CSS:

```css
.replay{position:absolute;top:18px;right:18px;background:rgba(212,165,116,0.1);
  border:1px solid rgba(212,165,116,0.3);color:#D4A574;
  font-family:monospace;font-size:10px;letter-spacing:.28em;text-transform:uppercase;
  padding:6px 12px;border-radius:1px;cursor:pointer;backdrop-filter:blur(6px);z-index:6}
```

## 4 · iframe embedding pitfalls (if cinematic is embedded in a deck)

### Pitfall 1 · Parent click-zone intercepts iframe-internal buttons

If the deck index.html has "left/right 22vw transparent click zones for paging", they will **cover the ▶ play button inside the iframe** — the user clicks the button but it's swallowed as "next page".

**Fix**: click zones add `top: 12vh; bottom: 25vh` — leaving the top and bottom 25% un-intercepted, so both the iframe's central ▶ and bottom-right ▶ are clickable.

### Pitfall 2 · iframe steals focus, keyboard events lost

After clicking the iframe, focus is in the iframe — the parent window can't receive ←/→ keyboard events.

**Fix**:
```js
iframe.addEventListener('load', () => {
  // Inject keyboard forwarder
  const doc = iframe.contentDocument;
  doc.addEventListener('keydown', (e) => {
    window.dispatchEvent(new KeyboardEvent('keydown', { key: e.key, ... }));
  });
  // Pull focus back to parent after click
  doc.addEventListener('click', () => setTimeout(() => window.focus(), 0));
});
```

### Pitfall 3 · file:// vs https:// behavior differences

A cinematic tested locally on file:// may break after deployment because:
- Under file:// iframe contentDocument is same-origin
- Under https:// it's also same-origin (if same host), but audio autoplay restrictions are stricter

**Fix**:
- Before deployment, use `python3 -m http.server` to test over local HTTP
- BGM must wait for the user click ▶ before `bgm.play()` — don't try to play on page load

## 5 · Anti-pattern quick-check

| ❌ Anti-pattern | ✅ Right pattern |
|---|---|
| Default = black screen ▶ overlay | Default = static dashboard, ▶ is auxiliary |
| 4 steps in a row, fade in on same screen | 5 scenes, full-screen switching, each focuses one thing |
| Reuse template + swap copy for different demos | Each demo has independent visual language (cover copy, can still tell apart) |
| Emoji / hand-drawn SVG as material | gpt-image-2 large image + extract_grid extraction |
| No BGM, no SFX | BGM + 11 SFX cues dual-track |
| setTimeout chain scheduling | requestAnimationFrame + global timeline T object |
| linear animation | Expo / cubic-bezier easing |
| No dev tools | `?seek=N` + `?autoplay=1` + REPLAY button |
| iframe button swallowed by parent click zone | click zone with top/bottom margin gives buttons room |

## 6 · Time budget

Following these patterns, one full cinematic demo (with dashboard):

| Task | Time |
|---|---|
| Design 5-scene narrative + visual language | 30 min (be careful — decides independence) |
| Dashboard static layout + content | 1 hour |
| Cinematic 5 scenes implementation | 1.5 hour |
| Audio cue timing + replay button | 30 min |
| Playwright screenshot validation of 5 key moments | 15 min |
| **Single demo total** | **3–4 hours** |

The second demo reuses the framework but **the visual language must be independent** — about 2–3 hours.
